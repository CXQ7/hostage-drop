# Hostage-Drop — 强干预型即时通讯实验场

一个基于 **STOMP/WebSocket + JMS + ActiveMQ** 的社交盲盒文件交换演示系统。

- **后端**: Spring Boot 3 · STOMP 网关 · JMS 持久化队列 · 动态 JmsListener 桥
- **前端**: Vue3 + Element Plus · @stomp/stompjs · 聊天气泡 UI · 文件上传 · 盲盒倒计时弹窗
- **中间件**: ActiveMQ 5.x (本机安装)

## 1. 目录结构

```
exp1/
├── backend/     Spring Boot 后端 (com.hostagedrop.gateway)
└── frontend/    Vue3 + Vite 前端
```

## 2. 启动顺序

### 第一步：启动 ActiveMQ

```bash
# 进入 ActiveMQ 安装目录的 bin 子目录
activemq.bat start
```

管理控制台: `http://localhost:8161`（默认账号 `admin/admin`）\
后端连接地址: `tcp://localhost:61616`

### 第二步：编译并启动后端

```bash
cd backend
mvn -DskipTests package
java -jar target/app.jar
```

> **注意**: Windows 中文路径下 `mvn spring-boot:run` 会出现主类加载异常，统一改用 `java -jar`。

后端启动后监听 `http://localhost:8080`，STOMP 端点为 `ws://localhost:8080/ws`。

### 第三步：启动前端

```bash
cd frontend
npm install      # 首次或新增依赖后执行
npm run dev
```

浏览器打开: `http://localhost:5173`

## 3. 通信架构

所有业务操作均通过 **STOMP over WebSocket** 完成，无 REST 接口。

### 客户端 → 服务端（发布）

| STOMP 目标      | 说明                       |
| --------------- | -------------------------- |
| `/app/initiate` | 发起盲盒担保交易           |
| `/app/pay`      | 支付赎金并请求释放真实文件 |

### 服务端 → 客户端（推送，用户私有）

| STOMP 目标           | 消息类型                   | 说明                     |
| -------------------- | -------------------------- | ------------------------ |
| `/user/queue/acks`   | `INITIATE_ACK` / `PAY_ACK` | 操作确认与仲裁结果       |
| `/user/queue/events` | `BAIT_NOTIFICATION`        | 接收方收到上锁盲盒通知   |
| `/user/queue/events` | `RANSOM_FILE`              | 发送方收到对方赎金文件   |
| `/user/queue/events` | `REAL_FILE_RELEASED`       | 接收方收到释放的真实文件 |
| `/user/queue/errors` | 错误                       | 网关处理异常             |

## 4. 关键设计

| 特性         | 实现方式                                                       |
| ------------ | -------------------------------------------------------------- |
| 用户身份绑定 | STOMP CONNECT 帧携带 `userId` header，后端提取为 `Principal`   |
| 点对点队列   | ActiveMQ 独立队列 `QUEUE_USER_<userId>`                        |
| 动态监听桥   | 用户上线时为其创建 `SimpleJmsListenerEndpoint`，下线时销毁     |
| 担保托管     | 真实文件写入 `QUEUE_ESCROW`，TTL 到期自动失效                  |
| 交易绑定     | `JMSCorrelationID = txId`，精准提取防误匹配                    |
| 绝对倒计时   | 服务端 `expireAt` epoch 毫秒推送给前端，前端 Watchdog 实时校准 |
| 超时失效     | 担保消息过期后 Selector 无法命中，pay 操作自然拒绝             |

## 5. 演示流程

1. **A、B 各自**在前端填写自己的用户 ID，点击「连接 STOMP 网关」
2. **A** 选择一张真实图片，填写接收方 B，点击「发起担保交易」
3. **B** 收到 `BAIT_NOTIFICATION`，弹出盲盒倒计时弹窗，聊天流中出现左侧气泡
4. **B** 在倒计时内切换到「支付赎金」tab，上传赎金图片，点击「支付赎金并请求释放」
5. **A** 收到 `RANSOM_FILE` 气泡（赎金图片）；**B** 收到 `REAL_FILE_RELEASED` 气泡（真实图片）
6. 倒计时结束或真实文件释放后，盲盒弹窗自动关闭

## 6. 注意事项

- 课程实验原型，文件内容以 Base64 Data URL 形式在 STOMP 消息体中传输
- 生产环境应补充：身份鉴权、消息签名防重放、文件分片存储、审计追踪、限流

## 7. 详细使用手册

### 7.1 角色说明

本系统模拟两个用户之间的文件"人质"交换。约定：

| 角色            | 说明                                                           |
| --------------- | -------------------------------------------------------------- |
| **发送方（A）** | 持有真实文件，将其托管至服务器作为担保，向对方发出上锁的"盲盒" |
| **接收方（B）** | 希望获得真实文件，须先提交赎金文件，仲裁通过后才能解锁         |

> 两个角色可以是同一台电脑的两个浏览器标签，也可以是两台不同的机器，只要都能访问同一个后端地址。

```
└──────────────────┴─────────────────────────────────────────┘
```

### 7.3 操作步骤（以 A 发送给 B 为例）

#### 第一步：双方各自连接网关

**A 的操作（浏览器标签 1）：**

1. 在「网关地址」填 `http://localhost:8080`（默认已填好）
2. 在「当前用户ID」填 `A`
3. 点击「连接 STOMP 网关」
4. 状态标签变为绿色「已连接为 A」即成功

**B 的操作（浏览器标签 2）：**

1. 同上，「当前用户ID」填 `B`
2. 点击「连接 STOMP 网关」，等待绿色状态

---

#### 第二步：A 发起盲盒（上传真实文件）

1. 点击底部「**发起盲盒**」Tab
2. 填写表单：

   | 字段      | 填写内容           | 说明                                |
   | --------- | ------------------ | ----------------------------------- |
   | 发送方ID  | `A`                | 与连接时的用户ID一致                |
   | 接收方ID  | `B`                | 要发给谁                            |
   | TTL（秒） | `180`              | 交易有效期，默认 3 分钟             |
   | 选择文件  | 点击按钮选一张图片 | 这是你想让 B 最终获得的**真实文件** |

3. 点击「**发起担保交易**」

**此时发生的事：**

- 真实图片被加密托管到服务器的 `QUEUE_ESCROW` 队列，设有 TTL
- B 的消息队列 `QUEUE_USER_B` 收到一条"盲盒通知"
- A 的聊天流右侧出现「你发起了一个加锁盲盒」气泡
- 聊天流中还会出현一条系统通知「担保已写入中间件，交易号 TX-xxx」

---

#### 第三步：B 收到盲盒通知

- B 的聊天流左侧出现「A 发来一个上锁盲盒」气泡
- 屏幕弹出盲盒弹窗，显示：
  - 发送方是谁
  - 文件名（内容被锁定，看不到）
  - 倒计时（绝对过期时间校准，每秒刷新）

> **注意**：弹窗仅作提示，无法从弹窗直接解锁。B 需要去「支付赎金」Tab 操作。

---

#### 第四步：B 支付赎金

1. 点击底部「**支付赎金**」Tab
2. 填写表单：

   | 字段     | 填写内容                   | 说明                            |
   | -------- | -------------------------- | ------------------------------- |
   | 交易ID   | 自动填入（从盲盒通知获取） | 也可手动填 A 那边的交易号       |
   | 付款方ID | `B`                        | 与连接时的用户ID一致            |
   | 收款方ID | `A`                        | 发送方                          |
   | 选择文件 | 点击按钮选一张图片         | 这是你愿意给 A 看的**赎金文件** |

3. 点击「**支付赎金并请求释放**」

**此时发生的事：**

- 赎金图片先发给 A（A 侧出现左侧气泡「对方发送了赎金文件」）
- 服务器从 `QUEUE_ESCROW` 中按 `txId` 提取真实文件，仲裁放行
- B 收到「真实文件已释放到你的会话」左侧气泡，图片可见
- A 收到「仲裁放行」系统通知
- 盲盒弹窗自动关闭

### 7.4 几种异常情况

| 情况                | 现象                                            | 原因                                          |
| ------------------- | ----------------------------------------------- | --------------------------------------------- |
| 倒计时归零后再支付  | B 侧弹出错误气泡「Escrow not found or expired」 | 担保消息 TTL 到期，已从队列中消失，无法再提取 |
| 交易ID 填错         | 同上                                            | `JMSCorrelationID` Selector 查不到对应消息    |
| 未连接就点发起/支付 | 弹出警告「请先连接 WebSocket 网关」             | STOMP 连接未建立，消息无法发送                |
| 未上传文件就点发起  | 弹出警告「请上传真实图片文件」                  | 本地校验拦截                                  |
| B 不在线时 A 发起   | B 再次上线后自动收到盲盒通知                    | JMS 持久化消息离线保存，上线时桥接推送        |

### 7.5 单人测试（同一浏览器）

如果只有一台电脑，可以用同一浏览器的两个**独立标签**分别模拟 A 和 B：

1. 标签 1：打开 `http://localhost:5173`，用户ID 填 `A`，连接
2. 标签 2：打开 `http://localhost:5173`，用户ID 填 `B`，连接
3. 在标签 1 发起盲盒 → 切换到标签 2 查看通知并支付 → 切回标签 1 确认结果
